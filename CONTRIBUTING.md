# Contributing Guidelines
## Rules Before Merging
All changes must be made through a separate branch. **Do not push directly to main**.
Every pull request must be reviewed before merging.
The project must run without errors before merging.
UI changes must not break existing pages or navigation.
Required features for that module must be implemented (incomplete work is allowed but should not break the app).
Keep commits meaningful and relevant to the change.
Do not merge if there are visible bugs, broken UI, or missing critical functionality.

## Code Standards
Follow consistent naming conventions across files, variables, and components.
Keep code clean, readable, and properly structured.
Avoid unnecessary complexity keep logic simple and understandable.
Do not hardcode values unless required for demo/mock purposes.
Maintain consistency in UI styling (colors, fonts, spacing).
Write modular code avoid large, unstructured files.
Ensure no unused code or console logs are left in the final version.
